# ResumeLoop

A **local-first, bring-your-own-AI** job-hunt tool. Paste a job description, get a tailored, ATS-optimized 1-page DOCX resume, a fit assessment, and outreach drafts — generated by **your own AI CLI**, on your own machine, with no API keys and no cloud account.

Your career data lives in a plain folder of markdown + JSON that you own and can `git` like any repo. The app is a fast local web UI (plus an optional terminal UI) over that folder.

> **Project status — migrating.** ResumeLoop v1 was a hosted, multi-tenant cloud app (AWS ECS + Neon + NextAuth). It is being re-platformed to the local-first model described here. The rationale and full decision set are in [ADR 0001](docs/adr/0001-pivot-to-local-first.md). The cloud build is frozen on branch `legacy/cloud-v1` (tag `v1.0-cloud-final`); see [`DEPRECATED.md`](DEPRECATED.md). The provider **spine** (`lib/providers/`) — the JD → fenced-JSON → DOCX round-trip — has landed; the larger cloud teardown and the npm CLI are in progress. Until then, run it from source (see [From source](#from-source)).

Try it at **[resumeloop.me](https://resumeloop.me)** — a free, anonymous demo backed by a small self-hosted model. Run it locally for full quality with the CLI you already pay for.

## Why local-first

- **No API keys.** The "brain" is the AI CLI you already have — `claude`, `codex`, `gemini`, or `opencode` — invoked in headless print mode. No per-token billing on top of your subscription, no key to store or encrypt.
- **No cloud, no account, no login.** The app binds to `127.0.0.1`. Your OS user account is the trust boundary. Nothing leaves your machine unless you choose to deploy it.
- **Your data is just files.** Profile, jobs, evaluations, and resumes are markdown/JSON in a folder you own. `git init` it, diff it, back it up, grep it. The database is a disposable cache rebuilt from those files.
- **Open-source engine, private data.** The repo ships the *generic* resume engine. Your actual profile and role preferences live only in your workspace — nobody forks a clone of you.

## What it does

- **Intake** job postings by pasting raw JD markdown, dropping `.md` files, or clipping with the Obsidian Web Clipper — each lands in `data/jobs/` as a markdown file with frontmatter.
- **Score** each job for fit percentage and role track (GenAI, Systems, SRE, IT-track) via your AI CLI.
- **Generate** a 1-page tailored ATS `.docx` per job through the strict generation harness (`buildv2.js`: char limits, verb-uniqueness, fit gate), plus a typographically polished PDF.
- **Track** the application pipeline (Saved → Applied → Phone Screen → Interview → Offer → Rejected → Ghosted) — stored in each job's frontmatter, indexed for fast funnel metrics.
- **Refine** bullets live in Chat with a side-by-side editor; import GitHub repos as context; review AI-suggested diffs before applying.
- **Outreach** — attach LinkedIn/alumni contacts per job; get AI contact summaries and draft emails + LinkedIn messages.
- **Onboard** in two phases: a deterministic wizard detects your installed CLIs and captures your details, then (once the brain is live) AI ingestion turns a pasted CV / LinkedIn / portfolio into a structured profile for you to confirm.
- **Profile editor** — drag-to-reorder experience, projects, and skills; toggle inclusion per entry; preview JSON diffs before saving.
- **Theme toggle**, **mobile + desktop responsive**, **interactive onboarding tour**.

## The brain: bring your own AI CLI

ResumeLoop does not call a model API directly. A **provider adapter** (`lib/providers/`) shells out to whichever AI CLI you configure, in headless/print mode, and parses its output:

| Provider | Invocation | Structured output |
|---|---|---|
| Claude Code | `claude -p --output-format json` | Native `.result` fast-path |
| Codex | `codex exec` | Fenced-JSON + Zod validate + one retry |
| Gemini CLI | `gemini -p` | Fenced-JSON + Zod validate + one retry |
| opencode | `opencode run` | Fenced-JSON + Zod validate + one retry |
| Ollama (or any OpenAI-compatible server) | `http` transport | Fenced-JSON + Zod validate + one retry |

Every structured prompt asks the brain to return a single ` ```json ` block matching a Zod schema. The adapter extracts the last fenced block, validates it, and retries once with the validation error on failure — so **no CLI is second-class**. Claude simply runs cleanest via its native JSON envelope.

The hosted demo at `resumeloop.me` uses the **same adapter** over its `http` transport, pointed at a small self-hosted model. No cloud key anywhere in the system.

## State model: files canonical, SQLite is a cache

```
data/                     # your workspace — git-trackable, you own it
  profile.json            # the single source of truth for resume content
  jobs/*.md               # JD + frontmatter (Action: 0–6, visa, apply_url, clipped_at)
  evaluations/*.md        # fit %, score, outreach brief
  resumes/                # generated .docx / .pdf

.cache/
  index.db                # rebuildable SQLite index over the files above
```

- **Files are authoritative.** Job text, tags, visa language, and pipeline stage live in the `.md` frontmatter. Profile content lives in `profile.json`.
- **The database is a query layer**, not a source of truth. It exists so the web UI can filter, aggregate, and draw funnel metrics over hundreds of jobs without re-parsing every file.
- **Resetting the DB loses nothing.** Delete `.cache/index.db` and a rescan rebuilds it from `data/`.

This inverts the old cloud model (DB-as-truth, multi-tenant). There is no `user_id`, no per-user isolation, no encrypted key storage — because there is one user and no keys.

## Quick Start

> Target distribution (npm CLI). Wiring is in progress; until it ships, use [From source](#from-source).

```bash
npm i -g resumeloop          # or: npx resumeloop
resumeloop init ~/career     # scaffold a git-trackable data/ workspace
resumeloop                   # boot the localhost web UI against it
resumeloop tui               # optional Ink terminal UI
```

On first run, Playwright auto-installs a headless Chromium (~150 MB) for HTML→PDF rendering. There is **no LibreOffice dependency** — DOCX is produced by the pure-JS `docx` package, and the polished PDF by a Playwright template.

### From source

```bash
git clone https://github.com/vietbui1999ru/ResumeLoop
cd ResumeLoop
npm install
npm run dev                  # http://localhost:3000
```

The first-run onboarding wizard detects your installed AI CLIs (`which claude codex gemini opencode`), lets you pick one, validates it with a test spawn, and captures your name/targets into the workspace. Then AI ingestion can build your profile from a pasted CV, a LinkedIn/portfolio URL, or a GitHub handle.

## First-time setup

1. **Pick your brain** — onboarding detects installed CLIs; choose Claude / Codex / Gemini / opencode (or point the `http` transport at a local ollama).
2. **Build your profile** — paste a CV / LinkedIn URL / GitHub handle; AI extracts `experience[]`, `projects[]`, `skills{}` into `profile.json` for you to confirm.
3. **Add jobs** — paste a JD, drop `.md` files, or clip with the Obsidian Web Clipper.
4. **Select jobs → Generate** — each produces a tailored `.docx` + `.pdf` in `data/resumes/`.

An interactive **onboarding tour** launches on first run and walks through each step; restart it any time from the Help menu.

## Generation engine

The ATS `.docx` comes from the strict harness carried over from the proven `resume-gen` engine:

- **`pipeline/buildv2.js`** — pure-JS `docx` generation. `T()` / `TL()` enforce per-bullet (≤116) and tagline (≤76) character ceilings on a word boundary.
- **`pipeline/master_resume_data.json`** — the engine's data shape (reborn as your workspace `profile.json`).
- **Validation harness** — verb-uniqueness across all 24 bullets, em-dash ban, 1-page paragraph budget, fit gate.
- **Playwright PDF** — an HTML→PDF template renders a second, typographically polished PDF. No system dependencies.

See [ADR 0001 §5](docs/adr/0001-pivot-to-local-first.md) for why LibreOffice was dropped.

## Engine vs data — the open-source split

The public repo ships **only the generic engine**: the validation mechanics, the role-track *schema*, the ATS keyword-bank *schema*, and the DOCX/PDF renderers. **All personal data** — your profile, role-track picks, keyword banks, verb-conflict overrides — lives in *your* `data/` workspace, seeded from blank templates and bootstrapped by the AI during onboarding.

The maintainer's real harness and data stay in a private workspace and on the frozen `legacy/cloud-v1` branch. They never ship in this repo.

## Architecture

```
You ──▶ localhost:3000 (Next.js, bound to 127.0.0.1)
            │
            ├─ app/(app)/   pages: jobs · chat · profile · settings
            └─ app/api/     local REST routes (no auth)
                  │
   Brain seam    ▼
   lib/providers/  ── provider adapter ── shells to your CLI / http transport
            │              (fenced-JSON + Zod validate + one retry)
            ▼
   Generation   buildv2.js (docx) ──▶ .docx
                Playwright template ──▶ .pdf
            │
   State    ▼
   data/*.{md,json}  (canonical)  ◀── rebuild ── .cache/index.db (query cache)
```

- **No middleware auth, no `user_id` filtering, no encrypted key store** — removed with the cloud platform.
- **Provider adapter** is the single seam serving local CLIs, the hosted demo, and local ollama — one place to test.
- **Optional Ink TUI** shares the Zod schemas, types, and file/index layer with the web app — one toolchain.

Deeper detail: [`docs/architecture.md`](docs/architecture.md) · invariants: [`CONTEXT.md`](CONTEXT.md) · one-screen overview: [`TLDR.md`](TLDR.md).

## Testing

```bash
npm test              # Vitest — unit + integration, once
npm test -- --watch   # watch mode
```

- **Adapter/spine tests** (`lib/providers/*.test.ts`) — inject a fake `CliRunner`, assert the fenced-JSON contract, retry behavior, and the JD → decision → DOCX round-trip. No real CLI needed.
- **DB-level tests** (`lib/*.test.ts`) — spin up an in-memory SQLite index via `initSchema`, wrap in `SqliteAdapter`, assert data invariants directly. No mocks.
- **Route tests** (`app/api/**/*.test.ts`) — call exported handlers directly, assert status + persisted writes.

Cross-browser E2E (Playwright) covers the web UI journeys.

## Tech Stack

- **Next.js 15** App Router + TypeScript — local web UI bound to `127.0.0.1`
- **Provider adapter** (`lib/providers/`) — shells to `claude` / `codex` / `gemini` / `opencode`, or an OpenAI-compatible `http` endpoint
- **Zod** — shared schemas for the structured-output contract (web + TUI)
- **SQLite** (`better-sqlite3`) — rebuildable local index; no cloud DB
- **docx** — programmatic ATS DOCX generation (`buildv2.js`)
- **Playwright** — headless Chromium for HTML→PDF, and cross-browser E2E
- **Ink** — optional terminal UI (React for the terminal)
- **gray-matter** — YAML frontmatter parsing for JD/profile files
- **Vitest** — unit + integration tests
