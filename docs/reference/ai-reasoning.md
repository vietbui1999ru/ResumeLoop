# AI Reasoning Pipeline

Given a raw job description, the reasoning pipeline calls Claude via tool use and returns a complete set of resume selections — track, work variant, work IDs, projects, tagline, persona title, and skills rows — ready to pass to `buildv2.js`.

## Entry Point

```ts
import { reasonForJob } from '@/lib/ai-reason'

const result = await reasonForJob(rawContent)
// result: ReasoningResult
```

`rawContent` is the full markdown body of the job posting (frontmatter stripped).

## Return Type

```ts
interface ReasoningResult {
  track:        string     // e.g. "Backend / API Engineer"
  workVariant:  string     // "genai" | "systems" | "IT-track"
  workIds:      string[]   // exactly 3: ["gitlab","carboncopies","udayton"]
  projects:     string[]   // exactly 3 project IDs
  personaTitle: string     // ≤60 chars
  tagline:      string     // ≤76 chars
  skillsRows:   string[]   // exactly 5: ["Tech · Tech · Tech", ...]
}
```

## How It Works

### 1. Build system prompt (`lib/prompt-context.ts`)

`buildSystemPrompt()` assembles the prompt by reading four sources from disk:

| Source | PATHS key | Content |
|--------|-----------|---------|
| `master_resume_data.json` | `PATHS.pipeline.masterData` | All bullets, project metadata, work experience |
| `ats-optimization-guidelines.md` | `PATHS.docs.atsGuidelines` | ATS formatting rules |
| `CLAUDE-full.md` | `PATHS.docs.claudeFull` | Role-track mapping table + hard constraints |
| `feedback/synthesized-rules.md` or `feedback/raw-log.md` | (see below) | Mistake history |

The prompt has four sections:

```
## Candidate Profile & All Bullet Data (master_resume_data)
<full JSON>

## Hard Constraints (MUST NOT violate)
<tagline ≤76, personaTitle ≤60, workIds exactly 3, ...>

## Role-Track Mapping & Work Variants
<CLAUDE-full.md content>

## ATS Optimization Guidelines
<ats-optimization-guidelines.md content>

## Mistake History — Avoid Repeating
<feedback content>
```

### 2. Call Claude (`lib/ai-reason.ts`)

Model: `claude-opus-4-7`  
Tool: `resume_decision` (forced via `tool_choice: { type: 'tool', name: 'resume_decision' }`)  
Max tokens: 2048

The tool schema enforces structure at the API level:

| Field | Schema constraint |
|-------|------------------|
| `workVariant` | `enum: ['genai', 'systems', 'IT-track']` |
| `workIds` | array of valid IDs, `minItems: 3, maxItems: 3` |
| `projects` | array of strings, `minItems: 3, maxItems: 3` |
| `personaTitle` | `maxLength: 60` |
| `tagline` | `maxLength: 76` |
| `skillsRows` | array of strings, `minItems: 5, maxItems: 5` |

### 3. Validate and auto-trim

`validateResult()` runs after the tool call returns:

| Check | Behavior |
|-------|----------|
| `workIds.length !== 3` | throws |
| `projects.length !== 3` | throws |
| `skillsRows.length !== 5` | throws |
| `tagline.length > 76` | auto-trims to last word boundary ≥ char 60; falls back to hard-slice at 76 |
| `personaTitle.length > 60` | auto-trims and strips trailing whitespace |

The tagline trim is intentionally lenient: Claude occasionally returns 77–80 chars despite the schema hint. The validator fixes silently rather than failing the whole call.

## Prompt Caching

The system prompt is sent with `cache_control: { type: 'ephemeral' }`:

```ts
system: [
  {
    type: 'text',
    text: systemPrompt,
    cache_control: { type: 'ephemeral' },
  },
]
```

**What this does:** Anthropic's prompt caching stores the processed system prompt in the KV cache for up to 5 minutes. Subsequent calls with the same system prompt bytes skip re-tokenizing and re-processing the large context (master data JSON + guidelines docs + role-track table).

**Why it matters for batch processing:** When you run AI reasoning against multiple JDs in sequence, every call after the first hits the cache. The system prompt is typically 4,000–8,000 tokens. Cache hits cost ~10% of normal input token pricing, so batch runs of 10+ JDs see significant cost reduction.

**Cache invalidation:** Any change to `master_resume_data.json`, `ats-optimization-guidelines.md`, `CLAUDE-full.md`, or the feedback files breaks the cache for that session. Feedback changes are common (the file is loaded on every call), so cache hits depend on feedback content being stable across a batch.

## Feedback Loop

`loadFeedbackContext()` selects the feedback source in priority order:

1. **`feedback/synthesized-rules.md`** — if present, use this. Generated by the `synthesize-mistakes` skill; concise distilled rules.
2. **`feedback/raw-log.md`** — if synthesized file absent, read last 10 `##`-delimited entries.
3. **Fallback** — returns the string `(no feedback history yet)`.

The feedback content is injected as the "Mistake History" section of the system prompt, so Claude can avoid repeating documented errors.

## Error Cases

| Error | Cause |
|-------|-------|
| `No tool_use block in AI response` | Claude returned text instead of a tool call. Should not occur with `tool_choice: { type: 'tool', name: 'resume_decision' }` — likely a transient API issue. |
| `Unexpected tool: <name>` | Tool name in response does not match `resume_decision`. |
| `workIds must have 3 entries, got N` | Claude returned wrong count despite schema constraint. Retry. |
| `projects must have 3 entries, got N` | Same. |
| `skillsRows must have 5 entries, got N` | Same. |

## Path Constants (`lib/paths.ts`)

All file paths used by the reasoning pipeline are centralized in `PATHS`:

```ts
PATHS.pipeline.masterData   // pipeline/master_resume_data.json
PATHS.docs.atsGuidelines    // docs/reference/ats-optimization-guidelines.md
PATHS.docs.claudeFull       // docs/reference/CLAUDE-full.md
PATHS.docs.atsSystem        // docs/reference/ats-optimized-resume-system.md
PATHS.docs.spec             // docs/reference/spec-job-match-resume-generator.md
```

`paths.ts` derives all paths from `process.cwd()` so they resolve correctly regardless of how the Next.js server is started.
